Next.js 15 캐싱 동작 정리

2026.04.18·8분 분량·324회 조회
#nextjs#caching#performance

Next.js 15 의 캐싱 동작은 14 와 비교했을 때 기본값이 꽤 달라졌습니다. 그 동안 "왜 새로고침해도 데이터가 안 바뀌지?" 라는 질문에 흐릿하게 답해 왔다면, 이번 글은 그 흐릿함을 걷어내기 위한 정리입니다.

결론부터 말하면 fetch · Route Handler · Server Component 는 서로 다른 캐시 계층에 얹혀 있고, 이걸 머릿속에 정확히 그릴 수 있느냐가 곧 디버깅 속도를 결정합니다.

캐싱 모델 한눈에

먼저 가장 많이 헷갈리는 fetch 부터 봅니다. App Router 에서 fetch 는 cache: 'force-cache' 를 명시적으로 주지 않는 한, 기본은 dynamic 입니다. 


        

        TypeScript
        
      
// app/posts/[slug]/page.tsx
export async function generateStaticParams() {
  const posts = await fetch('https://api/posts', {
    next: { revalidate: 3600 },
  }).then(r => r.json());

  return posts.map((p) => ({ slug: p.slug }));
}

revalidate 는 ISR 의 갱신 주기로 동작하지만, 동시에 같은 호출이 여러 컴포넌트에서 일어났을 때 하나의 in-flight 요청으로 묶이는 dedupe 기능도 함께 켜집니다.

다음 섹션에서는 같은 fetch 가 Route Handler 안에서 호출됐을 때 동작이 어떻게 달라지는지 살펴봅니다.
